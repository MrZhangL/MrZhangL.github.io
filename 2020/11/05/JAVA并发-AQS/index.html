JAVA并发-AQS

AQS是JDK在JUC中提供的一个抽象类，全称为AbstractQueuedSynchronizer，主要是通过继承的方式来使用，它本身没有实现任何的同步接口，仅仅是定义了同步状态的获取以及释放的方法来提供自定义的同步组件。它提供了一个FIFO队列，可以看成是一个用来实现同步锁以及其他涉及到同步功能的核心组件，我们经常使用的ReentrantLock就是通过AQS来实现的。

该篇文章也是对AQS的一个总结，因为AQS本身是一个抽象类，不会举例如何使用，因此适合了解过ReentrantLock等JUC源码的人阅读。

概述

根据AbstractQueuedSynchronizer的名字可以看出，它是一个基于队列的同步器，首先AQS是一个同步器，同步器的作用是用于控制共享资源的访问，在多线程竞争资源时，决定获取资源的规则，其次AQS是基于队列的，当某个线程竞争到资源时，其余的资源都进入队列进行等待资源被重新共享出来，同步器的主要特点如下：

  • 用 state 属性来表示资源的状态（分独占模式和共享模式），子类需要定义如何维护这个状态，控制如何获取锁和释放锁
    • getState()-获取状态
    • setState()-设置状态
    • compareAndSetState - 利用 cas 机制设置 state 状态
    • 独占模式是只有一个线程能够访问资源，而共享模式可以允许多个线程访问资源
  • 提供了基于 FIFO (First-In, First-Out)的等待队列，类似于 Monitor 的 EntryList
  • 条件变量来实现等待、唤醒机制，支持多个条件变量，类似于 Monitor 的 WaitSet

我们可以看出AQS的思想与Synchronized的思想非常类似，但是synchronized是基于C/C++实现的，并且是一个关键字，程序员无法对其进行扩展利用，缺乏灵活性，而AQS是基于Java实现的一个抽象的同步器，具有高度的扩展性与定制化，JUC包下的大部分工具类都是基于AQS实现的。

如果需要利用AQS来实现自定义的锁，可以使用AQS抽象类中的大多方法，AQS中有些方法的实现只有一句抛出 UnsupportedOperationException，我们应该重写(有些可以重写有些可以不重写，例如需要使用AQS到accquire方法就需要重写tryAcquire)：

  • tryAcquire：尝试获取锁
  • tryRelease：尝试释放锁
  • tryAcquireShared：尝试获得锁(共享模式)
  • tryReleaseShared：尝试释放锁(共享模式)
  • isHeldExclusively：是否持有独占锁

其他的很多方法AQS提供了实现(这些都是JDK提供的)：

  • acquire：阻塞获取锁
  • acquireInterruptibly：可打断的阻塞获取锁

等等。。还有很多方法

AQS核心成员

(1) int state

state是用于标志同步器状态的int变量，在线程竞争共享资源时，会根据state来判断自己能否竞争成功。

  • 例如在ReentrantLock中state变量用于记录同步器是否被占用，0为未被占用，大于0为被占用(重入锁将大于1)。
  • 例如在ReentrantReadWriteLock中state变量，高16位用于记录读锁的状态，低16位用于记录写锁的状态。

(2) Thread exclusiveOwnerThread

exclusiveOwnerThread用于记录独占该同步器的线程

  • 如果该同步器被某个线程独占，那么其他的线程无法获取到该同步器。反之，如果同步器没有被独占，这意味着可能可以有多个线程同时能获取到同步器，这取决于具体的实现。
  • exclusiveOwnerThread还用于锁重入的情况，通常在锁重入时(竞争该同步器)，会判断当前线程是否是独占该同步器的线程，如果是便能够获取到同步器。

(3) Node head, tail

可以看出，这个FIFO队列是通过链表的方式来实现的，当同步器被占用时，其他线程无法获取到同步器，那么其他的线程就会进入等待队列中，每个等待的线程用链表一个节点表示。每个节点Node保存了:

  • thread：等待的线程

  • waitStatus：等待的状态，节点等待的状态只能是以下几种

    • **CANCELLED(1)**：表示该节点已经被打断，无需进行Unpark了。

    • SIGNAL(-1)**：表示该节点的后继节点处于被阻塞的状态(被park或马上要被park了)，当该节点被释放时(即unlock)，应该unpark它的后继节点**，一般队列中的头结点不代表任何线程，如果它存在后继节点，那么头结点的状态则为-1，每次realese都是realese头结点，unpark它的后继节点则为最先加入等待队列的线程节点。下图是ReentrantLock的等待队列：

      等待队列中共有3个线程等待，头结点的状态为-1，第二个、第三个节点的状态也为-1，这是因为如果头结点被释放，第二个节点将变为头结点(只需要把Thread引用置为null即可)，这样新的头结点的状态也是-1。一般而言，如果一个节点的后继节点处于阻塞状态，那么这个节点的状态就为-1，而没有后继节点或后继节点不处于阻塞状态则该节点状态为0。

    • **CONDITION(-2)**：表示该节点在Condition的等待队列中。

    • PROPAGATE(-3)

    • 0：节点在等待队列中没有特殊含义

  • pre, next：前驱节点，后继节点

  • nextWaiter：用于Condition中保存wait的等待队列的后继节点指针，在Condition中等待队列为单向链表。

AQS核心方法

独占式获取同步器

public final void acquire(int arg)

该方法是用于线程来独占式地获得同步器的方法，例如当ReentrantLock在调用lock()方法时，就会去获取同步器，如果获取成功了，一般会设置线程独占然后直接返回，如果获取失败便会阻塞。

从方法的内容可以看出，首先会调用tryAcquire来尝试获取锁，如果获取失败了，再将线程放入等待队列中并进行阻塞。其中tryAcquire是一个抽象方法，需要子类去实现来定义什么情况下算是获取同步器失败。

而addWaiter和acquireQueued是AQS中已经实现的方法，并且都是不能被子类改写的，这两个方法定义了线程加入阻塞队列的一系列操作。

addWaiter方法

addWaiter将在链表尾部添加一个当前线程的节点，为了防止向尾部插入节点多线程的不安全性，利用CAS来对链表进行操作，如果CAS不成功会进入enq(node)进行反复尝试CAS操作直到加入链表成功。

endq同时还处理了尾结点为空的情况，那么证明这个链表中没有元素head和tail均为空，那么就会先利用CAS来设置头结点(设置好头结点后尾结点与头结点相同)，然后再CAS插入当前线程的节点：

acquireQueued方法

在该方法中：

  • 判断前驱结点是否是头结点。
    • 如果是头结点则再尝试一次加锁(自旋一次，因为是前驱节点是头结点，该节点就极有可能在同步器释放时获取到同步器，而如果前继节点不是头结点，那么就算同步器被释放，该节点也很难获取到同步器，应该是由它前面的先获得同步器)，加锁成功后就将这个节点变为头结点，将节点元素置为null，并返回不是被打断，即不是没有经过阻塞，而是自旋获取到锁的。
    • 如果不是头结点，则不需要进行自旋。
  • 如果前驱结点不是头结点或者是头结点但是自旋加锁失败，就检查该节点竞争锁是否应该阻塞(shouldParkAfterFailedAcquire)，在shouldParkAfterFailedAcquire方法中：
    • 如果前驱节点p的状态为-1就返回true应该阻塞，进入parkAndCheckInterrupt()进行阻塞
    • 如果前驱节点p的状态为0就将其置为-1，返回false，然后会再次进入上面的for循环，可能再进行一次自旋加锁，再到这里的时候就会判断前驱节点状态为-1，进入parkAndCheckInterrupt()阻塞。

独占式释放同步器

public final boolean release(int arg)

由上面地分析我们可以知道，当某个线程独占地获取到同步器后，会导致其他线程无法获取到同步器而被park阻塞住。当独占同步器的线程访问完共享资源后，应该主动释放同步器以至于其他的线程能够访问共享资源。同步器的释放定义在release方法中，用于释放独占同步器。

和前面的一样，tryRelease是一个抽象方法，交给子类去实现，当释放成功后，就会唤醒头节点的后续节点，在unparkSuccessor中：

该方法就是去唤醒头结点的后继节点，但是需要注意的是，等待队列中的节点线程并不一定是处于阻塞状态的(就算处于阻塞状态，也不一定是正在等待该同步器)，因为阻塞上锁是可以打断或超时的，ReentrantLock在上锁时就支持打断与超时，当在竞争同步器阻塞时，如果被打断或超时了，线程将不会阻塞，而执行其他的逻辑。对于这一类的节点，在取消阻塞时会将节点状态变为CANCELLED(1)。

基于以上这种情况，如果出现头结点的后继节点的状态为CANCELLED时，那么将会从链表的尾部向前遍历，找到最前面的节点状态<=0的节点进行唤醒，只要节点的状态<=0，它就是等待被唤醒的。

节点的线程被唤醒，并不意味着它一定能够获取到同步器，只是从他被阻塞的地方继续执行了，由上一小节的分析我们知道等待队列中的节点阻塞在acquireQueued方法中：

等待队列中节点线程阻塞在16行，当其被唤醒时，首先会标记interrupted为true，然后继续这个for循环，回到第7行去尝试获取同步器(竞争同步器)，如果竞争到同步器了，就会将当前节点设置为头结点。

注意：为什么唤醒的节点需要竞争同步器呢？因为在Thread0释放同步器时，Thread2被唤醒了，但是此时还有可能其他没有在等待队列中的线程也来竞争同步器了，会出现竞争的情况，如果Thread2没有竞争同步器成功，那么等待队列不会发生变化，这是一种非公平锁，我们也可以通过实现tryAcquire方法来实现公平锁。

共享式获取同步器

public final void acquireShared(int arg)

该方法是用于线程来共享式地获得同步器的方法，例如当ReentrantReadWriteLock的读锁WriteLock在调用lock()方法时，就会去获取同步器，如果获取成功了，就会直接返回，并不独占该同步器，这意味，可以有多个共享式的线程可以同时获取到同步器，如果获取同步器失败(同步器被独占时，共享式获取会失败)便会阻塞。

与acquire方法是类似的，先尝试获取共享地获取同步器，如果获取失败，再加入等待队列然后进行阻塞。其中tryAcquireShared也是一个抽象方法，由子类来实现获取同步器失败的逻辑。而doAcquireShared则是AQS已经实现了方法：

第3行给队列添加了一个新的节点，该节点代表当前线程，而与在acquire方法不同的是，acquire的执行的是addWaiter(Node.EXCLUSIVE)：

  • Node.EXCLUSIVE = null
  • Node.SHARED = new Node()

addWaiter传入的Node是作为新的线程的节点的nextWaiter，在独占式的获取同步器时，新加入等待队列节点的nextWaiter为null，而在共享式获取同步器时，新加入等待队列节点的nextWaiter为一个空的节点，在同步的等待队列中，这个nextWaite用于表示该节点是否为共享同步器线程。

6-22行与acquireQueued方法逻辑是类似的，先如果该节点的前驱节点是head，则自旋获取同步器，如果没有获取到同步器，就进入阻塞状态。如果该节点能够获取到同步器，则在acquireQueued新加了一些操作。注意，如果阻塞了被打断后会再尝试获取锁。

如果获取到了同步器，进入11~17行，有两种情况会进入：

  • 在获取同步器时进行自旋获取到了，没有经过阻塞状态
  • 进入阻塞状态后被打断，即被唤醒，又循环一次回到tryAcquireShared

当是第二种情况时，该共享式节点的后续可能还有有多个节点，需要注意的是，如果后继节点也是共享式节点，那么也要在这一次唤醒中都被唤醒，唤醒后继的共享节点通过setHeadAndPropagate方法实现：

  • 3行将本节点设置为头节点，并把本节点的thread置为null，即等效于将本节点移出队列
  • 8行判断s是否为共享节点，共享节点判断标志位nextWaiter是否为null，如果不为null代表是共享节点
  • 第9行执行共享节点的释放doReleaseShared，释放当前共享节点。doReleaseShared的具体代码到后面再分析，其主要作用就是unpark第一个节点(除头节点)的线程，通过这样的方式，又一个共享节点被unpark，则又会执行以上相同的逻辑，这样的unpark会一直传递下去，直到碰到的节点不是共享式的节点。

共享式释放同步器

public final boolean releaseShared(int arg)

该方法为共享同步器在释放同步的逻辑，它的实现与独占式的释放类似，但需要注意的是由于同步器是共享的，该方法可能被多个线程同时访问，因此在释放锁时需要注意多线程同步的问题。

和其他的方法一样，tryReleaseShared方法也是一个抽象方法，需要子类去实现的，例如在ReentrantReadWriteLock的ReadLock中，当所有的读锁都释放完毕后tryReleaseShared便返回true，然后在执行doReleaseShared。doReleaseShared主要执行的就是唤醒等待队列中线程。

条件变量

在AQS还有一个内部的公有类，叫ConditionObject，它是Condition接口的实现类：

注意：ConditionObject是AQS的一个内部类，并不是AQS的成员变量。

条件变量的作用是用于线程间的通信，即与在传统使用synchronized时，锁对象的wait与notify作用相似。Condition提供了await与signal方法与之对应，但是AQS中的Condition使用比传统的更加灵活。AQS可以创建出多个Condition对象，不同的Condition对象之间互相不干扰，而传统的wait与notify只能通过被锁对象进行调用，因此在notify时，所有wait的线程都可能受到影响。
